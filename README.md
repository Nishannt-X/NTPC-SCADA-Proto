# NTPC Lara Real-Time Telemetry Platform

A fully working, event-driven, horizontally scalable distributed backend simulating real-time power plant telemetry monitoring for NTPC Lara (2×800MW thermal units).

All sensor data is **synthetic** — generated by the system itself. No real hardware involved. Every tool is free/open-source and runs locally.

## Services

### 1. `sensor-simulator` (port 8081)
Generates synthetic sensor readings for **30 virtual sensors** (15 per unit) covering 5 types: TEMPERATURE, PRESSURE, VIBRATION, RPM, LOAD. Uses **random-walk with mean-reversion** for realistic drift. Supports **fault injection** via REST API.

### 2. `ingestion-service` (port 8082)
Kafka consumer (`ingestion-group`) that buffers readings and persists them to TimescaleDB in **batched inserts** (50 readings or 1s flush interval). Handles malformed messages gracefully.

### 3. `anomaly-detection-service` (port 8083)
Independent Kafka consumer (`anomaly-detection-group`) that evaluates readings against configurable thresholds. Requires **2 consecutive breaches** before firing an alert (noise resistance). Publishes to `sensor-alerts` topic.

### 4. `alerting-service` (port 8084)
Consumes alerts, **deduplicates** (same sensor+severity within 5 min → update only), persists to DB, and logs **stubbed notifications**. Includes **auto-resolution** of stale alerts.

### 5. `query-api` (port 8080)
REST API for the frontend. Reads from TimescaleDB + Redis cache. Proxies fault injection to the simulator.

## Quick Start

### Prerequisites
- Docker & Docker Compose
- Java 17 & Maven (for local dev without Docker)

### Option 1: Docker Compose (Full Stack)
```bash
# Clone and start everything
docker compose up --build -d

# Check all containers are running
docker compose ps

# View logs
docker compose logs -f query-api
```

### Option 2: Infrastructure Only + Local Services
```bash
# Start infrastructure
./infra-up.sh

# Run individual services (in separate terminals)
cd sensor-simulator && mvn spring-boot:run
cd ingestion-service && mvn spring-boot:run
cd query-api && mvn spring-boot:run
cd anomaly-detection-service && mvn spring-boot:run
cd alerting-service && mvn spring-boot:run
```

### Option 3: Kubernetes (Kind)
```bash
cd k8s
./deploy.sh
```

## API Endpoints

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/v1/units/{unitId}/readings/latest` | Latest reading per sensor (cached 5s) |
| `GET` | `/api/v1/units/{unitId}/readings?from=&to=&sensorType=&page=&size=` | Historical readings with pagination |
| `GET` | `/api/v1/alerts/active` | All unresolved alerts |
| `GET` | `/api/v1/alerts/history?from=&to=&unit=&severity=` | Historical alerts with filters |
| `POST` | `/api/v1/simulator/inject-fault` | Trigger fault injection |

### Example curl Commands

```bash
# Latest readings for Unit 1
curl http://localhost:8080/api/v1/units/UNIT_1/readings/latest | jq

# Historical readings (last hour)
curl "http://localhost:8080/api/v1/units/UNIT_1/readings?from=2024-01-15T09:00:00Z&to=2024-01-15T10:00:00Z" | jq

# Active alerts
curl http://localhost:8080/api/v1/alerts/active | jq

# Inject a fault (SEVERE temperature spike on Unit 1)
curl -X POST http://localhost:8080/api/v1/simulator/inject-fault \
  -H "Content-Type: application/json" \
  -d '{"unit":"UNIT_1","sensorType":"TEMPERATURE","magnitude":"SEVERE","durationCycles":10}'
```

## How to Demo

### 1. Start the system
```bash
docker compose up --build -d
```

### 2. Verify data is flowing
```bash
# Check latest readings (should show 15 sensors for UNIT_1)
curl http://localhost:8080/api/v1/units/UNIT_1/readings/latest | jq '. | length'
# Expected: 15
```

### 3. Trigger a fault and watch the alert pipeline
```bash
# Inject a SEVERE temperature fault on Unit 1 (10 cycles ≈ 30 seconds)
curl -X POST http://localhost:8080/api/v1/simulator/inject-fault \
  -H "Content-Type: application/json" \
  -d '{"unit":"UNIT_1","sensorType":"TEMPERATURE","magnitude":"SEVERE","durationCycles":10}'

# Wait ~10 seconds, then check for alerts
curl http://localhost:8080/api/v1/alerts/active | jq
```

### 4. Watch the logs
```bash
# See the anomaly detection fire
docker compose logs -f anomaly-detection-service | grep "ALERT FIRED"

# See the alerting service deduplicate and notify
docker compose logs -f alerting-service | grep "NOTIFY"
```

### 5. View metrics in Grafana
Open http://localhost:3000 → "NTPC Lara Telemetry — Overview" dashboard.

## Anomaly Thresholds

| Sensor Type | WARNING | CRITICAL | Unit |
|---|---|---|---|
| TEMPERATURE | > 560 | > 580 | °C |
| PRESSURE | > 185 | > 195 | bar |
| VIBRATION | > 0.5 | > 0.7 | mm |
| RPM | > 3150 | > 3250 | rpm |
| LOAD | > 92 | > 97 | % |

## Architectural Highlights

- **Event-driven**: All inter-service communication via Kafka — no REST calls between services
- **Multi-consumer pattern**: `sensor-readings` topic consumed independently by ingestion-service AND anomaly-detection-service (different consumer groups)
- **Batched writes**: Ingestion buffers readings for efficient batch inserts (50 rows per flush)
- **Noise-resistant detection**: 2 consecutive threshold breaches required before alerting
- **Alert deduplication**: Same sensor+severity within 5 min → update only, not duplicate
- **Auto-resolution**: Stale alerts resolved automatically when threshold is no longer breached
- **Redis caching**: Latest readings endpoint cached with 5s TTL — measured ~10x faster
- **Horizontal autoscaling**: HPAs on ingestion-service and query-api (1→5 replicas at 60% CPU)
- **Schema migrations**: Flyway-managed, versioned SQL — no `ddl-auto`
- **Full observability**: Every service exposes `/actuator/prometheus`, scraped by Prometheus, visualized in Grafana
